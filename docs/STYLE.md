# 代码风格

## 代码哲学（Robert C. Martin）

以下原则来自 Robert C. Martin 的《Clean Code》，用来收紧整个仓库的代码质量态度。它们影响注释怎么写、函数怎么拆、命名怎么起、以及合入前要问自己的问题。

### 核心态度

**童子军原则**：让代码比你发现时更整洁——每次签出都比签入时好一点点。不只「写完就跑」，每次顺手把附近的不整洁修掉再合入。

**遵循标准惯例**：团队应当基于行业通用规范约定一套编码标准，并在代码中保持一致。具体规范不重要，重要的是所有人遵循同一套规范。代码本身就是最好的标准文档，不需要另写一本规范手册。

**不要随意**：结构代码的每个决定都应有理由，并且这个理由应当通过代码结构本身传达出来。如果结构看起来是随意的，别人就会觉得可以随便改。如果结构在全系统中保持一致，别人就会遵循这个惯例继续使用它。

**精确**：代码中的每个决定都要精确地做出。知道你为什么要这样决定、如何处理异常情况。不要懒于做精确的决定。如果调用可能返回 null 的函数，确保检查 null。如果需要处理货币，用整数或专门的 Money 类并正确处理舍入。如果有并发更新的可能，确保实现某种锁机制。

### 函数设计

**函数的单一职责**：一个函数只做一件事，做好，做唯一。如果无法用一句简洁、具体的话描述函数做了什么事，说明它该拆了。

**短函数**：函数应该很短，通常不超过 20 行。如果一个函数不能在一个屏幕内读完，它就是太长了。短函数让分支、错误路径和副作用更容易被看到。

**函数只下降一层抽象**：函数中的语句应当全部在同一抽象层级上，且该层级应比函数名所描述的操作低一层。混杂不同抽象层级是阅读的最大敌人。重构时当你按抽象层级拆分函数，常常会发现原本被掩盖的新的抽象层级问题。

**无副作用**：函数不应做其名称未承诺的事情。调用者依据函数名建立预期，违背预期就是 bug。

**命令查询分离**：函数要么执行命令（写状态），要么回答查询（读状态），不应同时做两件事。

**用多态替代条件判断**：遇到 switch/case 或 if/else 链时，优先考虑多态。一条规则：同一类型的选择在系统中最多只出现一次 switch，该 switch 必须用于创建多态对象，之后所有同类选择靠多态分派。

**封装条件**：布尔逻辑本身已经够难懂了，别再把它们塞在 if 或 while 语句中间。抽取解释条件意图的函数。例如 `if (shouldBeDeleted(timer))` 胜于 `if (timer.hasExpired() && !timer.isRecurrent())`。

**避免否定条件**：否定式比肯定式更难理解。尽可能把条件表达为肯定形式。例如 `if (buffer.shouldCompact())` 胜于 `if (!buffer.shouldNotCompact())`。

**标识参数令人困惑**：向函数传入布尔值是一种恶劣做法。它立即让方法签名变得复杂，宣告本函数不止做一件事。标识为 true 时做一件事，false 时做另一件事。应该拆分为两个独立的函数。

**选择器参数**：几乎没有什么比函数调用末尾那个悬空的虚假参数更令人憎恶。它把多个函数合并成了一个。选择器参数是懒于把一个大函数拆分为多个小函数的借口。不管是布尔值、枚举还是整数，都应该拆分为多个函数。

**防止太多参数**：最理想的参数数量是零，其次是一，再次是二，应尽量避免三。三参数以上就需要非常特殊的理由——而且即便如此也不该用。参数越多，测试组合越多，可读性越差。

**避免输出参数**：输出参数反直觉。读者期望参数是输入而非输出。如果函数必须修改某物状态，让它修改所属对象的状态。

**死函数**：从不被调用的函数应当删掉。保留死代码是浪费。不要害怕删除函数。源代码控制系统会记住它。

**理解算法**：不要满足于测试通过。在认为一个函数完成之前，确保你理解它是如何工作的。光靠凑 if 语句和标志位让代码「能跑」是不够的。重构函数至清晰表达的程度，让算法如何工作显而易见。

### 命名规则

**命名即文档**：好的命名不需要注释来解释。如果需要一个注释来补充名字的含义，那这个名字就没有体现其意图。先改名字，再考虑加注释。

**选择描述性名称**：不要急于定名。确保名称具有描述性。记住，含义会随软件演化而漂移，所以经常重新评估名称的恰当性。软件中 90% 的可读性来自命名。花时间明智地选名，并保持其相关性。

**在恰当的抽象层级选择名称**：不要选用透露实现细节的名称；选择反映函数或类所在抽象层级的名称。每次审阅代码时，你都会发现一些命名层级过低的情况。抓住机会修改它们。

**尽可能使用标准命名**：基于已有约定或惯例的名称更易理解。例如使用 DECORATOR 模式时，应在装饰类名中使用 Decorator 一词。使用团队内通用语言中的术语。命名中携带越多的项目相关语义，读者就越容易理解代码在说什么。

**无歧义的名称**：选择让函数或变量的工作方式毫无歧义的名称。如果函数名只有宽泛模糊的描述，说明它不合格。

**长名称用于长作用域**：名称长度应与其作用域大小对应。作用域只有 5 行时，`i` 和 `j` 完全没问题。但跨越大段距离时，短名称的含义会丢失。作用域越长，名称就越要长而精确。

**避免编码**：名称中不应编码类型或作用域信息。`m_` 或 `f_` 之类前缀在现代开发环境中毫无用处。项目或子系统编码如 `vis_` 既干扰又冗余。让名称摆脱匈牙利语标记法的污染。

**名称应描述副作用**：名称应描述函数、变量或类的一切所作所为。不要用名称来隐藏副作用。不要用一个简单动词来描述做了不止一件简单动作的函数。例如 `getOos()` 实际上也会创建 OOS，所以应命名为 `createOrReturnOos`。

### 代码结构

**除去重复**：相同逻辑只有一份责任归属。被复制的每一份都是一颗定时炸弹。最明显的重复是复制粘贴的代码块，应替换为简单方法。更隐蔽的重复是反复出现的 switch/case 或 if/else 链，应替换为多态。再进一步，是拥有相似算法但不共享代码行的模块，应使用 TEMPLATE METHOD 或 STRATEGY 模式。

**三振出局**：第一次写下去，第二次忍住复制，第三次就开始抽离。不要过早抽象，但也不容忍第三份重复。

**死代码**：死代码是不会被执行的代码——在不可能触发的 if 语句体中、在从不抛异常的 catch 块中、在从不被调用的小工具方法中、在永不发生的 switch 分支中。代码会随着设计变化而未被完全更新，它仍然能编译但不遵循新的约定或规则。发现死代码时，给它一个体面的葬礼，从系统中删除。

**不一致性**：如果你用某种方式做一件事，就用同一方式做所有类似的事。这回到最小惊讶原则。小心选择你遵循的约定，选定后务必持续遵守。如果在某个函数中用 `response` 表示 `HttpServletResponse`，就在其他使用 `HttpServletResponse` 对象的函数中保持一致。

**杂乱**：无实现的默认构造器有何用？它徒然用无意义的废料搞乱代码。未使用的变量、从不调用的函数、不添加信息的注释等等，都是杂乱，应移除。保持源文件干净、组织良好、无杂乱。

**意图模糊**：我们想要代码尽可能具有表达力。冗长的表达式、匈牙利标记法和魔术数字都会掩盖作者的意图。花时间让你的代码意图对读者可见。

**隐藏时序耦合**：时序耦合往往是必要的，但不应隐藏它。结构化函数参数以使调用顺序显而易见。用桶式传递（bucket brigade）暴露时序耦合：每个函数产生一个后续函数需要的值，这样就不可能以错误的顺序调用它们。

**将逻辑依赖改为物理依赖**：如果一个模块依赖另一个模块，这种依赖应该是物理的而非逻辑的。依赖模块不应做假设（逻辑依赖），而应明确向被依赖模块询问所需的所有信息。例如不要硬编码页大小常量，而是调用格式化器提供的 `getMaxPageSize()` 方法。

**封装边界条件**：边界条件很难跟踪。把边界条件的处理放在一个地方。不要让 `+1` 和 `-1` 散落各处。例如如果 `level + 1` 出现两次，就应封装到 `nextLevel` 变量中。

**用命名常量替代魔术数**：原始数字出现在代码中通常是个坏主意。把它们藏在命名良好的常量后面。魔术数不仅指数值，也指任何值不能自我描述的记号。例如 `assertEquals(7777, ...)` 中的 7777 就是魔术数，应改为 `assertEquals(HOURLY_EMPLOYEE_ID, ...)`。

**避免依恋情结**：一个类的方法应当对该类自己的变量和函数感兴趣，而不是对其他类的变量和函数。当一个方法使用另一个对象的访问器来操作该对象内部数据时，它就是在依恋那个类的范围。消除依恋情结，因为它将一个类的内部暴露给另一个类。

**避免传递性导航（得墨忒耳律）**：我们不想让单个模块了解太多其协作者的信息。如果 A 与 B 协作，B 与 C 协作，我们不想让使用 A 的模块知道 C。`a.getB().getC().doSomething()` 这种链式调用应避免。改为让直接协作者提供所有需要的服务。

**用结构替代约定**：用结构而非命名约定来强制设计决策。命名约定虽好，但不如强制遵守的结构。例如带有良好命名的枚举的 switch 不如带有抽象方法的基类——没有人被迫每次都按同一方式实现 switch，但基类确实强制要求具体类实现所有抽象方法。

**将可配置数据放在高层**：如果存在像默认值或配置值这样在高抽象层级已知和预期的常量，不要把它埋在低层函数中。将其作为参数从高层函数传入低层函数。配置常量应位于非常高的层级，易于修改。

### 错误处理

**错误处理是一件事**：错误处理是正常逻辑之外单独的一件事。函数不应把错误处理混在主路径里——提前 return、及早检查错误码、把错误路径和正常路径在视觉上分开。

**先写 Try-Catch-Finally 语句**：编写可能抛出异常的代码时，先用 try-catch-finally 语句定义范围。这帮助你定义无论 try 块中发生什么错误，调用者都应预期的行为。用 TDD 方式：先编写强制异常的测试，然后在处理器中添加行为满足测试。

**使用异常而非返回错误码**：返回错误码搞乱调用者代码，调用者必须立即检查错误且容易遗忘。改为抛出异常，调用代码更整洁。另外，错误码枚举是「依赖磁铁」——每次新增错误都需要重新编译所有引用该枚举的模块。

**不要传空，不要返空**：函数不接收 NULL，也不返回 NULL。返回 NULL 就是把责任推给调用者。如果必须表示缺失，用 Rust 的 `Option`、C 的返回 `bool` + 输出参数或明确的有意义哨兵值。向方法传 NULL 更糟糕——在大多数编程语言中都没有好的方法来处理调用者意外传入的 null，恰当的做法就是禁止传入 null。

**定义常规流程（特例模式）**：用特例模式（SPECIAL CASE PATTERN）避免让异常打断业务逻辑。创建一个类或配置一个对象，让它为你处理特例，这样客户代码就不必处理异常行为。

### 测试

**驱动代码测试**：干净的代码是经过充分测试的代码。没有测试的整洁只是表演。

**测试三定律**：第一定律：在编写通过测试的生产代码之前，先编写失败的单元测试。第二定律：不要编写超过足以失败程度的单元测试（编译失败也算失败）。第三定律：不要编写超过足以通过当前失败测试程度的生产代码。

**FIRST 原则**：测试应当快速（Fast）、独立（Independent）、可重复（Repeatable）、自验证（Self-Validating）、及时（Timely）。

**每个概念一个测试**：每个测试函数应测试一个单一概念。不要在一个函数中测试多个不相关的事。每个断言的数目应当最小化，但更重要的是每个概念只有一个断言组。

**测试边界条件**：特别注意测试边界条件。我们常常把算法中间部分搞对，却错估了边界。

**测试应快**：慢的测试是不会被运行的测试。当进度紧张时，首先被去掉的就是慢测试。所以尽一切可能让你的测试保持快速。

### 覆盖率忽略规则

以下情况可以用 `#[cfg_attr(coverage, coverage(off))]` 从覆盖率统计中排除：

**1. 测试代码**

每个测试模块前都应标记：

```rust
#[cfg(test)]
#[cfg_attr(coverage, coverage(off))]
mod tests { ... }
```

**2. FFI 边界**

对 C 侧（`c_vp_*`）的调用无法在单元测试中执行。如果函数仅包含 FFI 调用及必要的数据转换，可以在函数上标记；如果整个模块所有公开函数都依赖 FFI，可以在文件头标记。

**3. Unsafe util**

使用了 `UnsafeCell` 等底层原语的工具模块。覆盖率工具无法完美的处理此类代码，因此即使编写了测试也可能无法反映在覆盖率报告上。测试仍然需要编写，但可以标记忽略。

## 注释风格

- 优先使用中文
- 尽量短，尽量不要句中换行，尽量不要尾随句号
- 优先解释「为什么这样做」「哪里不能乱动」「什么情况下会出问题」
- 能从命名和类型直接看出来的事实尽量不再写
- 如果觉得需要注释来解释一段代码，优先考虑重构，而不是直接加注释
- 代码变更后，过时的注释就是谎言——注释不会自动同步

### 更适合保留的注释

- 中断与主循环的职责边界
- 所有权、可重入性和调用上下文限制
- 状态机为什么这样收敛
- 为什么这里要退避、缓存、延迟或丢弃一次尝试
- 芯片、协议栈或硬件行为带来的特殊约束

### 应尽量避免的注释

- 逐行翻译代码在做什么，或重复函数名/变量名已表达清楚的事实
- 把开发过程、试错记录写进源码注释
- 本该进入长期文档的跨模块规则只留在局部注释里
- 注释掉的代码、署名归属、变更日志等元数据（源代码控制系统会记住它们）
- 过时的、与当前代码状态不符的注释（精确的谎言比空白更可怕）

## 日志风格

### 日志格式

Rust 统一使用 `log` crate，C 统一使用仓库内日志宏。

统一输出格式：

- Rust：`[LEVEL] [module] message`
- C：`[LEVEL] [C:module] message`

约定：

- `LEVEL` 只使用 `ERROR`、`WARN`、`INFO`、`DEBUG`
- Rust 不直接绕过 `core/src/utils/logger.rs` 自己拼串
- Rust 模块名默认来自 `module_path()`，但做可读压缩：去掉 crate 前缀，尾部 `::mod` 折叠，根模块显示为 `core`
- C 不直接新增裸 `PRINT(...)`，优先使用 `VP_LOG_ERROR`、`VP_LOG_WARN`、`VP_LOG_INFO`、`VP_LOG_DEBUG`
- C 侧 `module` 使用稳定短名，例如 `main`、`runtime`、`ble_gap`、`ble_hid`
- message 优先使用稳定短句；带参数时使用 `message;key=value,key2=value2` 风格，尽量短，优先小写英文，避免感叹号和无意义省略号
- 一条日志里优先带上最关键的状态字段，例如 `reason=%u`、`handle=%u`、`wired_active=%u`

### 级别语义

| 级别 | 用于 | 不用于 |
|--|--|--|
| `ERROR` | 明显错误、调用时序错误、不变量被破坏、panic、关键失败 | 正常但少见的状态、纯 bring-up 观察信息、仍在预期内的状态切换 |
| `WARN` | 非预期但系统能继续运行（降级、回退、重试、丢弃、stub 被调用） | — |
| `INFO` | 低频且关键的生命周期里程碑。只在默认日志阈值下值得长期保留的重要事件 | 高频路径默认不要打 `INFO` |
| `DEBUG` | 状态变化细节、输入变化、运行时决策细节。只在调试时有价值、长期看会嫌多的信息 | — |

**高频路径规则**：高频路径默认不打印。只在稳定状态变化、首次异常、显著回退或限频摘要时打日志。不打印每个样本、每个 queue push/pop、每次 poll、每次 EXTI 原始电平。

**典型文案模板：**

| 场景 | 文案 |
|--|--|
| 初始化完成 | `core initialized` |
| 状态切换 | `usb state changed;state=...,wired_active=...`、`route changed;route=usb` |
| 连接建立 | `connected;handle=...` |
| 调用时序错误 | `api misuse;func=vp_core_poll,reason=before_init` |
| panic | `panic;info=...` |
| unsupported stub | `feature unavailable;feature=power_enter_sleep` |

如果某条规则会被多个模块长期复用，应优先改成统一日志文案，再继续扩散到新代码。

### 什么时候该打日志

**更适合保留：**

- 初始化完成或失败
- route、USB、BLE、power 等关键状态切换
- 会影响后续收敛路径的异常分支
- unsupported stub 被调用
- 硬件 bring-up 阶段的重要条件变化
- 数据被丢弃、忽略、回退、重试等影响诊断判断的事件
- API 在错误时机被调用

**应尽量避免：**

- 高频路径逐次打印（每帧姿态、每次轮询、每个 debounce tick、每个 EXTI 原始电平）
- ISR、主循环、定时器回调里的流水账式日志
- 能从上层已有日志稳定推出的重复日志
- 只有「到过这里」价值、没有状态信息的日志
- 未来长期不会用于定位问题的临时调试输出

**写日志前的自问清单**：在新增日志前，先问自己：

1. 这条日志在长期运行里值得保留吗？
2. 如果它每秒出现 100 次，我还愿不愿意看？
3. 出问题时我会不会真的靠它定位？
4. 它是不是只是为了证明代码走到了这里？

如果答案偏第 4 条，这条日志通常不该进入正式代码。

## Git commit message 风格

### 格式

强制使用单行，不使用 scope，不写 body：

```
<type>: <summary>
```

示例：

```
feat: 增加按键边沿检测
fix: 恢复 ble 重连流程
refactor: 拆分板级输入层
docs: 重写 ffi abi 说明
chore: 对齐 c_api 格式与 clangd 配置
```

### type 约定

- `feat`：新增功能、能力或对外行为
- `fix`：修复错误、修正异常行为
- `refactor`：重构、整理结构，不改变预期对外行为
- `docs`：文档、注释、说明文字调整为主
- `chore`：构建、脚本、配置、格式化、仓库维护杂项
- `test`：测试相关

如果一个提交同时像 `feat` 又像 `refactor`，优先按主要目的写。

### summary 规则

- 用动词开头，直接说改了什么，不说空话
- 优先写结果和对象，不写主观评价
- 尽量控制在一行内
- 一个 commit 只做一件事；如果写不清，优先拆 commit，不是补 scope
- 如果后一个 commit 只是修前一个 commit 的遗漏，合并前优先 squash
- 不要把功能、重命名、格式化、文档混进同一个 commit

**正确示例：** `feat: 实现 usb 与 ble 路由切换`
**避免**无信息量词语：`cleanup`、`better`、`update`、`wip`、`misc`
**避免**空泛：`重构一下`、`更新文档`、`修复 bug`、`临时提交`

### 语言规则

- `type` 固定使用英文小写
- `summary` 强制使用中文
- 术语、模块名、协议名、文件名等确实没有自然中文说法时，可保留必要英文名词
- 不使用 scope
