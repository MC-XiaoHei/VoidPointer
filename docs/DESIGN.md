# VoidPointer 设计总览

产品目标、硬件背景和总体分层架构。实现状态和任务拆分见 `TASKLIST.md`。

## 1. 产品目标

VoidPointer 是一款集成空中鼠标、激光指示和便携存储能力的三模演示器。

几个核心目标：

- 轻巧便携，替代掉「翻页笔 + U 盘」的组合
- 待机时间长，避免充电成为负担
- 使用方便，符合使用者直觉
- 使用与配置无需安装驱动

## 2. 硬件设计

### 2.1 核心元件选型

- 主控：`CH585`
- IMU：`LSM6DSV`
- USB Hub：`CH334`
- TF 读卡器：`GL823K`
- 电池管理：`ETA6005`
- DC-DC：`TPS63100`

### 2.2 USB 设计

设备带有可拔插的尾插：尾插拔下时露出 Type-A 公口直连主机，插回时恢复 Type-C 母口，免去携带转接线的麻烦。

为了提高读写性能，降低主控负担与功耗，设备采用了内置 Hub，将 TF 读卡器和主控分别连接到 Hub 的不同端口上，从而实现了独立的读写路径。

### 2.3 按键布局

设备一共有五个物理按键：

- 正面从前到后：Context、Action，以及一个跷跷板结构的 Up/Down
- 背面从前到后：Primary、Secondary

握住设备时，无论左右手，拇指正好落在 Action 键上，食指落在 Primary，中指落在 Secondary。

每个按键的功能可通过配置中的按键映射表调整，单个键最多可同时映射为四个逻辑功能，支持用侧面三档物理开关切换三组不同的映射配置（详见 3.6）。

### 2.4 电源边界

设备没有设计物理总电源开关，完全依赖电源状态机管理。

状态机设计有三个电源状态：

- `Active`
- `Suspend`
- `Sleep`

详细行为见 `POWER_STATE_MACHINE.md`

## 3. 固件设计

### 3.1 架构设计

为了更好的开发体验与漏洞管控，我们采用了 Rust + C 的双语言固件开发模式。

C 负责硬件事实和平台动作，不直接访问产品业务状态；Rust 负责业务状态和决策，不直接碰寄存器或协议栈细节。

两种语言通过自动生成的 FFI 接口进行互操作，并统一编译，避免手动维护 FFI 接口带来的风险。

### 3.2 中断与主循环

所有输入事件通过 ISR 上报，主循环负责统一消费。

IMU 中断只负责唤醒，实际 FIFO 读取由 Rust 按需发起。

主循环不通过 GPIO 轮询制造业务事件。

### 3.3 输入模型

物理按键通过消抖后，根据当前按键映射表转换为逻辑功能。

映射表存储在配置中，支持用物理开关切换三个配置（详见 3.6）。

- 上/下按键为跷跷板结构，互斥，通过映射表产生 scroll 事件
- 所有按键使用二态输入，平台层负责在状态变化后重新设置下一次中断

### 3.4 IMU 模型

- `LSM6DSV` 中断只负责唤醒，不承载姿态数据语义
- FIFO 读取由 Rust 决定是否发起
- 姿态数据采用 latest-sample 缓存，不保留每一帧历史

### 3.5 指针模式

按住 Action 键时进入空中鼠标模式。设备相对基准姿态的角度偏移映射为指针移动。

指针映射有两种模式，可通过配置切换：

- 位置映射：角度直接映射为屏幕坐标偏移。符合直觉，适合需要精确定位的场景。
- 速度映射：角度映射为光标速度。偏转角度越大，光标移动越快，松开按键后立即停止，不保留惯性。适合需要大幅度移动的场景。

### 3.6 配置切换

设备支持三组独立的配置，通过物理开关切换。

每组配置独立保存以下内容：

- 按键映射表（每个物理键映射到哪些逻辑功能）
- 可选的指针配置，例如灵敏度，翻转 X/Y 轴等

同时，指针配置也拥有一份全局配置，若独立配置未设置对应的配置，则使用对应的全局配置。

这意味着用户可以在三组配置中分别设定适合不同场景的映射方案（例如演示模式、设计模式、媒体控制），通过开关即时切换。

## 4. 路由策略

支持三条路由：BLE、2.4G、USB。

当前默认采用 USB 优先策略：

- USB 进入 `Configured` 后，鼠标报告只走 USB，同时关闭无线广播并断开现有连接
- USB 退出 `Configured` 后，恢复无线广播与无线待机能力

详细状态规则见 `ROUTE_STATE_MACHINE.md`

## 5. 设备配置

为了避免安装驱动，采用了基于浏览器的 WebHID 技术对设备进行配置。

- 用户无需安装平台专用软件
- 配置保存在设备 DataFlash，跟设备走，不跟主机走
- 协议细节见 `VENDOR_PROTOCOL.md`
- 存储格式见 `CONFIG_SPEC.md`
